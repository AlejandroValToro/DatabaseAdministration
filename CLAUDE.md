## Project Overview

This is a database administration academic project implementing a **parking lot theft report system** (*Sistema de Denuncias de Robos en Parqueaderos Públicos de Medellín*) on **Oracle 19c**.

The repository contains two main artifacts:
- `script_parqueaderos_oracle.sql` — Full Oracle SQL script (DDL, DML, queries, PL/SQL objects)
- `modelo_documento.md` — Academic document with ER model, relational model, and normalization analysis

## Running the Script

The script is self-contained and idempotent. Execute it in SQL*Plus or SQL Developer against an Oracle 19c schema:

```sql
-- In SQL*Plus:
SET SERVEROUTPUT ON;
@script_parqueaderos_oracle.sql
```

The script handles its own cleanup (Section 1 drops all objects before recreating them), so it can be re-run safely.

To test the stored procedure in isolation:
```sql
SET SERVEROUTPUT ON;
EXEC sp_escalar_denuncias(300000);
```

To invoke the function directly:
```sql
SELECT fn_valor_total_denuncia(1) FROM DUAL;
```

## Script Structure

The SQL script is organized into 5 sections:

1. **Cleanup (DDL)** — Drops all tables and PL/SQL objects with `CASCADE CONSTRAINTS`, silencing errors if they don't exist
2. **Table creation (DDL)** — Creates 10 tables with constraints, check constraints, and surrogate keys via `GENERATED BY DEFAULT AS IDENTITY`
3. **Test data (DML)** — Minimum 4 rows per table, designed so all queries return non-empty results; ends with `COMMIT`
4. **Queries** — 4 queries demonstrating: multi-join (5 INNERs), LEFT JOIN, set operation (MINUS), and correlated subquery
5. **PL/SQL objects** — One function, one trigger, one stored procedure, followed by verification queries

## Database Schema

The domain models theft reports at public parking lots. Key design decisions:

- **CELDA** is a weak entity: surrogate PK `id_celda` + UNIQUE constraint on `(id_parqueadero, numero_celda)`
- **DENUNCIA** uses a total-disjoint specialization: `tipo_denuncia` discriminator (`'PENAL'`/`'INFORMAL'`) + two subtype tables (`DENUNCIA_PENAL`, `DENUNCIA_INFORMAL`) sharing the same PK as FK to the supertype
- **VEHICULO_PROPIETARIO** is a junction table for the N:M relationship between vehicles and owners
- **LOG_CAMBIOS_DENUNCIA** is an audit table populated exclusively by the trigger

## PL/SQL Objects

| Object | Type | Purpose |
|---|---|---|
| `fn_valor_total_denuncia(p_numero_denuncia)` | Function | Returns total value of stolen objects for a given complaint; returns 0 if none found |
| `trg_log_cambio_tipo_denuncia` | Trigger | AFTER UPDATE on `DENUNCIA.tipo_denuncia` — logs old/new type, date, and `USER` to `LOG_CAMBIOS_DENUNCIA` |
| `sp_escalar_denuncias(p_umbral)` | Procedure | Iterates informal complaints via cursor; if total stolen value exceeds threshold, upgrades to penal (updates type, inserts into `DENUNCIA_PENAL`, deletes from `DENUNCIA_INFORMAL`). Uses `SAVEPOINT` per row for individual atomicity; commits at end |

The procedure intentionally chains all three objects: it calls the function, which triggers the trigger on update.

## Naming Conventions

- Table and column names: `UPPER_SNAKE_CASE`
- Constraints: prefixed by type (`pk_`, `fk_`, `uk_`, `ck_`)
- PL/SQL: functions prefixed `fn_`, procedures `sp_`, triggers `trg_`
- Parameters in PL/SQL: prefixed `p_`, local variables `v_`
